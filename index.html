Progressive Enhancement

After reading Aaron Gustafso article I think i progressive enhancement as a content-first strategy for web development. Instead of building a site exclusively for the newest browsers and hoping it works everywhere, you start with a solid foundation of core content marked up in HTML. Using Gustafson's Peanut M&M analogy this core content is the "peanut". From there, you progressively layer on the "chocolate" of CSS for visual styling, and finally add the "hard candy shell" of JavaScript for rich interactivity. If a user's browser or device cannot process the CSS or JavaScript they still are able to access the information without the page breaking.

This is super important in web development because the web is accessed on a wide range of devices and networks. JavaScript can easily fail to load due to network drops, caching, or errors. By treating advanced functionality as an optional enhancement rather than a strict requirement, devs ensure that websites remain accessible and fast. It prevents a "blank white screen" scenario and makes sure that everyone receives a minimum viable experience.

Semester and Project Reflection

Reflecting on my experience in IFSC 1310 this semester I found the transition into front-end web development to be a rewarding challenge. Because my professional background is in IT infrastructure and vulnerability management I am used to looking at systems from a security and administration perspective. Getting into HTML and CSS required me to shift gears and think about user experience, accessibility, and visual structure. What I think i've done particularly well this semester was grasping the strict syntax and validation requirements. Structuring semantic HTML5 felt logical to me.

If I had to start this project or the semester over, I would definitely approach file pathing and scope differently from the beginning. Early on, I fell into the trap of using absolute paths like /mars-rovers/assets which caused major headaches when thinking about deploying to GitHub Pages. I also initially tried to over-engineer my site by throwing in advanced accessibility and CSS pseudo-elements before I mastered the core concepts. If I were to do it again, I would stick strictly to relative paths (`./`) from day one and focus purely on getting the standard semantic tags to do the heavy lifting before adding any extra bells and whistles.

The feedback I received on my project drafts was incredibly helpful in course-correcting. Specifically, it was pointed out that I had a broken image path for the Perseverance rover, inconsistent email links in my footer, and markup that was too complex. To address these issues, I went back through my directory structure and standardized all my links to relative paths to guarantee they wouldn't break. I also chose to chnage feedback regarding code complexity. I stripped out unnecessary tags and screen-reader skip links. While I know those are important for modern web development, removing them allowed me to make sure I accomplished all the task in a functional webpage without extra fluff.
